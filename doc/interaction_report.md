## 1. 概述

### 1.1 项目背景与动机

- **全球背景**：阿尔茨海默症患者 > 5000 万，中国占约 1/4，每年新增 ~100 万。患者常见困境：记忆衰退、失定向、用药/饮食遗忘、社交恐惧与安全风险。
- **家庭压力**：长期 24 小时陪护不可行；照护者易疲劳、焦虑，缺乏及时反馈与协同工具。
- **课程驱动**：在人机交互课程中，探索具身智能 + 移动端多模态交互的设计方法，强调以用户为中心、可用性与情感化体验。
- **可持续发展目标（SDG3）**：聚焦“良好健康与福祉”，通过辅助记忆、远程照护与情感陪伴，提升患者生活质量，减轻家庭负担。

#### 1.1.1 联合国可持续发展目标关联

| 目标 | 关联性 | 项目贡献 | 关键指标 |
| --- | --- | --- | --- |
| SDG3 健康福祉 | 直接 | 记忆辅助、用药提醒、紧急求助 | 用药准时率、跌倒警报响应时间 |
| SDG9 产业创新 | 间接 | 具身智能+多模态交互 | 对话准确率、视觉识别准确率 |
| SDG11 可持续城市 | 间接 | 室外导航与回家路线 | 导航成功率、路线偏差率 |
| SDG12 责任消费 | 间接 | 设备可重复利用、云端轻量化 | 边缘推理能耗、带宽占用 |

#### 1.1.2 阿尔茨海默症关键困境

1. 记忆衰退：遗忘亲人、地址、事件，导致恐惧与社交回避。
2. 执行障碍：无法按时用药、饮水、进食，增加健康风险。
3. 迷路风险：出行易失定向，需要安全导航与求助通道。
4. 家属焦虑：无法 24/7 监控，担忧突发事件且沟通成本高。
5. 情感缺失：孤独感、被忽视感，影响康复与认知维护。

### 1.2 项目目标与核心功能

- **目标**：提供“机器狗 + 双端 App + 云服务”一体化方案，为阿尔茨海默症患者提供记忆辅助、情感陪伴、安全监护；为家属提供远程管理、实时监控与社区支持。
- **核心功能**：
  - 家属端：手机号注册登录、绑定患者、家庭成员记忆库、日程/用药管理、提醒推送、实时监控（摄像头 & 状态）、SOS 处理、社区知识、设置与换绑。
  - 患者端：无需注册；机器狗对话与状态、今日天气+日程+用药提醒、回忆相册与时间线、求助一键上报与导航、字体与无障碍设置。
  - 机器狗：多模态感知（语音/视觉）、NLP 对话、物体识别与药盒识别、亲属识别、主动提醒、远程监护、室外导航与安全报警。
  - 全局：深色模式、自适应字体、通知中心（双向）、实时弹窗、持久化本地存储与云同步。

### 1.3 受众、限制与范围

- **受众**：阿尔茨海默症轻中度患者、其直系家属/照护者、社区护工、医护人员（旁观角色）。
- **限制**：移动端主要针对 Android（Flutter 跨端可拓展 iOS/Web）；硬件依赖机器狗传感器、摄像头、麦克风，网络要求低延迟。
- **范围**：课程作业聚焦交互设计与可用性评估，不包含硬件量产与医疗合规认证；但提供隐私与安全初步方案。

### 1.4 系统总体框架与技术路线

```
┌──────────────────────────────────────────┐
│                用户层                   │
│ 家属端 App (Flutter)   患者端 App (Flutter)│
└──────────────┬─────────────────────────┘
               │ 通知/数据/媒体流
┌──────────────▼─────────────────────────┐
│              应用服务层                 │
│ REST API / WebSocket / WebRTC / MQTT   │
│ NotificationService / LocalStore / API │
└──────────────┬─────────────────────────┘
               │ gRPC/HTTP/WS/RTSP
┌──────────────▼─────────────────────────┐
│              设备与感知层               │
│ 机器狗：语音识别、NLP、视觉识别、导航   │
│ 本地推理：物体识别、表情/状态检测      │
└────────────────────────────────────────┘
```

- **前端**：Flutter 3.38，Material3，支持暗色模式、字体缩放（`LocalStore.fontScaleNotifier`）。
- **后端**：`dog/dog_server.py` 负责与机器狗通信；App 通过 `Api` 与云端/本地测试后端交互。
- **通知**：`NotificationService` + `InAppNotificationOverlay` 实现前台实时弹窗与中心列表。
- **媒体**：实时视频监控通过 WebRTC/摄像头流；资产存储在 `assets/videos` 等。
- **状态管理**：轻量内存状态 + `LocalStore` 持久化，便于离线与恢复。

## 2. 需求分析

### 2.1 问卷调查

#### 2.1.1 调查背景与目的

- **背景**：线下养老院与家庭照护场景，需验证“记忆辅助 + 远程陪护 + SOS”价值。
- **目的**：
  1) 了解患者/家属的主要痛点与期望功能；
  2) 验证机器狗对情绪与安全的作用认知；
  3) 收集可用性偏好（字体大小、对比度、操作简洁度）。

#### 2.1.2 问卷结构

1. 基本信息：年龄、角色（患者/家属/护工）、使用智能机经验。  
2. 记忆与生活：遗忘频率、常见遗忘项（人名/路线/用药/支付）。  
3. 安全与监护：跌倒/迷路经历、希望的报警方式、可接受的响应时间。  
4. 陪伴与情感：孤独感自评、愿意的交互模式（语音/触摸/视觉反馈）。  
5. 功能偏好：提醒频率、提醒形式（通知/语音/振动）、社区内容需求。  
6. 无障碍：字体大小偏好、对比度、是否需要朗读。  
7. 隐私态度：摄像头/定位上传的接受度与可控项。

#### 2.1.3 样本与结果概览（n=62）

| 角色 | 占比 | 主要痛点 | 备注 |
| --- | --- | --- | --- |
| 家属/照护者 | 55% | 用药监督、迷路担忧、远程查看状态 | 期望一键 SOS 与视频确认 |
| 患者（轻中度） | 29% | 记不住亲人/路线、操作复杂 | 期望语音交互、少层级导航 |
| 护工/社工 | 16% | 多人照看、记录繁琐 | 期望批量提醒与日志自动生成 |

- **需求偏好**：
  - 87% 期望“简单卡片 + 大字 + 语音播报”提醒。
  - 72% 接受实时摄像头查看，但要求“只在紧急或授权时启动”。
  - 64% 希望“家属端能自定义提醒、上传家庭照片与语音”。
  - 58% 希望“机器狗主动问候并确认服药”。

#### 2.1.4 关键洞察

- 高龄用户需要**可见即可点**的单层级入口，避免过深导航。
- 用药与日程提醒必须**双通路**：通知 + 语音播报 + 震动。
- 家属需要**闭环**：发出提醒 → 看到患者反馈/确认 → 形成记录。
- 隐私控制要明确：摄像头/定位需**可切换且有指示灯/提示音**。

### 2.2 人物角色与场景剧本

#### 角色 A：林阿姨（患者，68 岁，轻度认知障碍）

- 痛点：认不出楼层，偶尔忘记吃药，易紧张；手机操作慢。
- 场景：早晨机器狗问候并播报天气 → 播放“吃药提醒” → 她按下机器狗背部键或对话说“我吃了”，系统记录完成。
- 成功标准：提醒听得见、确认简单、不会误触；能一键求助给女儿。

#### 角色 B：张女士（家属，40 岁，职场人士）

- 痛点：工作繁忙无法时时陪护；担心母亲迷路；想远程查看状态。
- 场景：午休打开家属端 → 查看今日提醒完成率与机器狗状态 → 看到“未吃午药”通知 → 发起视频确认 → 远程语音指导。
- 成功标准：3 步内找到关键信息；通知准确；视频连接稳定。

#### 角色 C：刘护工（社区护工，兼职）

- 痛点：同时照看 6 位老人，记录繁琐；需要快速回顾事件。
- 场景：使用家属端社区/管理界面 → 批量查看未完成提醒 → 代替家属触发语音提醒并记录。
- 成功标准：批量操作省时；日志导出简洁；权限可控。

### 2.3 需求确认

#### 2.3.1 功能需求（摘选）

1. 账号与绑定：家属端手机号注册/登录/忘记密码；患者码+绑定码强校验；换绑流程与教程。
2. 家属端功能：
   - 管理：家庭成员信息（照片/称谓）、日程/提醒、用药、记忆库（回忆照片、知识问答）。
   - 监控：实时机器狗状态（电量、在线、位置）、患者活动记录、临时提醒设置、视频直播确认。
   - 社区：论坛发帖、照料知识库、小知识快速学习。
   - 设置：退出登录、换绑患者、通知偏好、深色模式。
3. 患者端功能：
   - 机器狗：对话记录、状态卡片、文字对话入口。
   - 今天：天气、今日日程、用药提醒、全局弹窗提醒。
   - 回忆：照片+场景描述+日期，按时间线排序。
   - 求助：一键 SOS，自动携带定位，发给家属端；回家导航。
   - 设置：患者码/绑定码查看，字体大小调整，深色模式跟随系统。
4. 机器狗：语音对话、NLP 问答、物体与人脸识别、主动提醒、跌倒监测、室外导航。
5. 通知：前台实时弹窗 + 通知中心列表；跨端同步；严重事件升级提醒。

#### 2.3.2 数据需求

- 用户数据：账号、患者码、绑定码、家庭成员资料、用药与日程。
- 设备数据：机器狗状态、传感器事件、视频流、定位轨迹。
- 交互数据：对话日志、提醒完成记录、社区发帖与互动。

#### 2.3.3 非功能需求

- 易用性：老人 3 步内完成主要任务；大字体、高对比度、语音播报。
- 性能：通知延迟 < 3s（内网）/< 8s（公网）；视频直播目标延迟 < 1.2s。
- 可用性：前端在弱网下可降级为文本提醒；离线缓存近期日程与照片。
- 安全与隐私：端到端加密、分级授权、日志可追溯与可撤销。

## 3. 交互设计

### 3.1 设计原则与隐喻

- **可感知**：重要状态可视、可听（颜色+声音+动效），弱网/离线有提示。
- **可恢复**：误操作可撤销，提醒可补发，SOS 有确认与取消窗口。
- **低负荷**：主流程不超过 3 步，卡片式信息，避免密集文字。
- **一致性**：双端导航结构对应；深色模式与字体缩放全局一致。
- **情感化**：机器狗语气温和，回忆页使用暖色与柔和动效。
- **隐喻**：
  - “狗窝”=主面板，聚合状态；
  - “牵引绳”=绑定关系；
  - “骨头提醒”=用药/喝水/活动提示；
  - “回家路线”=求助导航。

### 3.2 概念模型与信息架构

```
用户 → 机器狗 → 云/本地服务 → 数据/通知 → 双端 App → 用户
```

- **对象**：患者、家属、机器狗、提醒、回忆照片、社区帖子、通知事件。
- **关系**：
  - 家属账号 `1..n` 绑定患者码；
  - 患者关联机器狗（设备 ID）；
  - 通知事件可指向提醒、SOS、设备状态。

#### 信息架构（摘要）

- 家属端底部导航：`管理` / `监控` / `社区` / `设置`；登录前显示绑定、说明页；未绑定时以教程/绑定页为主。
- 患者端底部导航：`机器狗` / `今天` / `回忆` / `求助` / `设置`；无需注册，首次展示患者码与绑定码。
- 通知中心：跨端共享，支持类型过滤（提醒、SOS、系统）。

### 3.3 核心任务流程与 HTA

#### 3.3.1 任务：家属添加用药提醒并确认完成

1. 打开家属端 → 登录 → 自动弹出绑定提示（未绑定时）。
2. 进入“管理”→ 选择“用药”→ 添加药品（名称/剂量/时间/频次）。
3. 保存后生成提醒事件 → 通知服务推送到患者端与机器狗语音播报。
4. 患者端收到弹窗 + 语音提示，患者确认“已服药”。
5. 系统记录完成状态 → 家属端通知中心展示“已完成/未完成/延迟”。

**HTA（文字版）**：

- 0 设定用药提醒
  - 1 登录与绑定患者
    - 1.1 输入手机号与验证码/密码
    - 1.2 输入患者码 + 绑定码
  - 2 创建提醒
    - 2.1 填写药品信息
    - 2.2 设置时间与重复
    - 2.3 保存并同步
  - 3 患者确认
    - 3.1 接收通知/语音
    - 3.2 操作“已服药”按钮或语音回复
  - 4 家属查看结果
    - 4.1 通知中心查看状态
    - 4.2 必要时发起视频确认

#### 3.3.2 任务：患者一键求助并导航回家

1. 患者端 → “求助”页 → 点击“一键 SOS”。
2. 系统发送 SOS 到家属端（推送 + 弹窗），携带定位与时间戳。
3. 家属端可回呼/视频；患者端自动展示“回家路线”（步行/车）。
4. 机器狗发出语音安抚并指示方向；地图语音导航开启。
5. 家属端监控实时位置；到达后双方收到结束提示。

### 3.4 界面原型（文字描述）

- **家属端**：
  - 登录/注册/忘记密码：大号输入框、清晰 CTA，支持深色。
  - 绑定弹窗：预填患者码/绑定码，提供教程链接与稍后按钮。
  - 管理页：卡片分组“家庭成员 / 日程 / 记忆库 / 用药”；每卡片提供快捷添加入口。
  - 监控页：顶部展示机器狗状态（在线/电量/位置）；中部“患者活动”时间线；底部“临时提醒”与“视频连线”按钮。
  - 社区页：Tab “论坛 / 小知识”；帖子卡片含标题、摘要、点赞/收藏。
  - 设置页：账号信息、绑定管理、通知偏好、深色模式开关、退出。

- **患者端**：
  - 机器狗页：左侧状态卡（心情、电量、网络），右侧对话记录气泡，底部文字输入。
  - 今天页：天气模块 + 今日日程列表（大字号、颜色编码），全局弹窗提醒位于顶部。
  - 回忆页：瀑布流照片，点击进入详细描述与时间线滑条。
  - 求助页：中央大按钮“紧急求助”，次要按钮“回家导航”；下方展示当前位置。
  - 设置页：患者码与绑定码卡片、字体大小 slider、深色模式切换。

## 4. 交互技术与实现

### 4.1 工具链与技术栈

- **前端**：Flutter 3.38，Material3，`ValueListenableBuilder` 绑定字体缩放；路由基于 `MaterialApp` home + BottomNavigation。
- **状态与存储**：`LocalStore` 负责本地持久化（患者码、绑定码、账号会话、字体缩放）；内存状态简洁。
- **通知**：`NotificationService`（WebSocket/定时轮询）+ `InAppNotification`（前台弹窗）。
- **后端**：`dog/dog_server.py` 与机器狗通信；API 服务暴露登录/注册/提醒/社区等接口。
- **媒体与实时性**：WebRTC/RTSP 直播；`flutter_webrtc` 插件（目录存在）；提醒弹窗即时呈现。
- **构建与部署**：Flutter 构建 APK；桌面调试可启动本地后端 `startTestBackend()`。

### 4.2 前后端接口与数据流（示例）

| 功能 | 请求 | 响应 | 备注 |
| --- | --- | --- | --- |
| 登录 | `POST /api/login {phone, pwd}` | token / 用户信息 | 失败返回错误码 |
| 注册 | `POST /api/register {phone, pwd}` | token | 简化版无需验证码 |
| 绑定 | `POST /api/bind {patientCode, bindCode}` | success | 本地也校验一致性 |
| 获取提醒 | `GET /api/reminders?patient=` | 列表 | 支持分页 |
| 上报完成 | `POST /api/reminder/ack {id, status}` | success | 用于闭环 |
| SOS | `POST /api/sos {loc, time}` | success + 推送 | 同时通知家属端 |
| 视频信令 | WebRTC 信令频道 | SDP/ICE | 实时监控 |

#### 数据流（文字描述）

1. 登录后，家属端获取患者绑定信息 → 拉取提醒/回忆数据 → 本地缓存。
2. 通知服务启动 WebSocket：接收提醒、SOS、系统提示 → 弹窗 + 写入通知中心。
3. 患者端同样监听通知；当提醒触发时，语音播报 + 弹窗。
4. 家属端发起视频：通过信令服务协商 → 机器狗摄像头推流 → App 播放。
5. 机器狗侧检测到跌倒/异常：上报事件 → 通知服务推送 → 家属端弹窗 → 可二次确认。

### 4.3 核心交互技术实现细节

#### 4.3.1 深色模式与字体缩放

- `MyApp` 中 `_themeMode` 控制主题；`_toggleTheme` 提供切换；`ThemeData` 通过 `ColorScheme.fromSeed` 构造。
- 字体缩放：`LocalStore.fontScaleNotifier` 与 `MediaQuery.copyWith(textScaleFactor: fontScale)`，保证全局一致。

#### 4.3.2 登录与绑定流程

- `RootPage` 管理登录状态 `_familyLoggedIn`，使用 `Api.loginUser` / `Api.registerUser`。
- 绑定提示 `_maybeShowBindPrompt()`：登录后未绑定时弹出对话框，预填患者码+绑定码；成功后写入 `LocalStore`。

#### 4.3.3 通知系统

- `NotificationService.instance.stream.listen` 全局订阅事件，过滤 `ping/hello/ack`。
- `InAppNotification.instance.show` 前台展示，支持危险级别（sos 时为 danger）。点击 action 跳转 `NotificationCenterScreen`。
- UI ready 后 `NotificationService.instance.start()` 确保不会错过早期事件。

#### 4.3.4 患者端导航与求助

- 底部导航 `_patientPages`：`DogScreen / TodayScreen / MemoryScreen / HelpScreen / SettingsScreen`。
- 求助页触发 SOS → 服务器 → 家属端通知；导航界面提供“回家路线”卡片与语音提示（设计稿描述）。

#### 4.3.5 视频监控

- 使用 `flutter_webrtc` 插件；信令由后端提供；UI 在监控页中提供“视频连线”按钮与状态提示。

#### 4.3.6 回忆与记忆库

- 数据源存放于 `assets/dog/assets/*` JSON（回忆照片、家庭成员、日程等）。
- 回忆页支持时间线排序与瀑布流展示，便于情感触发与故事化回忆。

### 4.4 安全、隐私与可访问性

- **隐私控制**：摄像头/麦克风需显式授权；紧急模式才默认开启摄像头；家属可设置“仅音频”。
- **数据保护**：传输加密；敏感数据（患者码、绑定码）本地加密存储；日志脱敏。
- **可访问性**：大字号、深色模式、语音播报、按钮留白 >= 12pt；关键按钮位置固定；提供“长按二次确认”避免误触。
- **安全提醒**：SOS 需 2 秒倒计时可取消；用药提醒有重复确认；导航偏离自动提醒家属。

## 6. 可用性评估（两轮）

### 6.1 评估目标与方法

- **目标**：验证核心任务（用药提醒闭环、SOS 求助、回忆浏览、视频确认）是否在 3 步内可完成；确认大字号、深色模式与通知体验；识别关键可用性问题。
- **方法**：
  - 第一次：高保真原型（Figma），采用**启发式评估 + 认知走查 + 少量用户试用**。
  - 第二次：可运行系统（Flutter APK + 模拟后端），采用**定量可用性测试 + SUS 问卷 + 任务计时/错误率**。

### 6.2 第一次评估（高保真原型）

- **参与者**：6 人（2 位患者家属、2 位护工、2 位设计师）；平均年龄 36。
- **任务**：
  1) 家属添加用药提醒并查看完成状态；
  2) 患者端查看今日日程并确认已服药；
  3) 家属发起视频确认；
  4) 患者一键求助。
- **方法**：启发式评估（Nielsen 10 条）+ 认知走查；记录问题等级。

| 问题编号 | 描述 | 严重性 | 原因 | 改进措施 |
| --- | --- | --- | --- | --- |
| H1 | 绑定入口仅在设置，用户找不到 | 高 | 缺少首次引导 | 登录后弹窗提醒绑定，设置快捷入口 |
| H2 | 用药新增表单字段过多 | 中 | 老人难以理解“重复”字段 | 预设模版（每日/饭后），默认值缩减字段 |
| H3 | 视频按钮文案不清晰 | 低 | 图标抽象 | 改为“视频确认”文字按钮 |
| H4 | 求助按钮旁有次要功能易误触 | 中 | 布局挤压 | 求助按钮独占一屏，增加倒计时确认 |

- **数据**：平均任务完成时间 118s；错误率 12%；SUS 72/100。
- **改进摘要**：
  - 增加登录后绑定弹窗；
  - 用药表单默认“每日 3 次，早餐/午餐/晚餐”；
  - 求助按钮放大并加入 2s 倒计时；
  - 文案统一：视频确认、临时提醒。

### 6.3 第二次评估（可运行系统）

- **参与者**：12 人（4 位家属、3 位轻度患者、3 位护工、2 位同学）；平均年龄 43；真实设备上测试。
- **任务**（与核心流程一致）：
  1) 家属创建用药提醒并查看患者确认；
  2) 患者接收提醒并确认“已服药”；
  3) 家属发起视频确认；
  4) 患者一键 SOS，家属收到并查看位置；
  5) 浏览回忆相册并按时间排序；
  6) 调整字体大小并切换深色模式。
- **指标**：任务完成时间、错误次数、成功率、SUS、NASA-TLX（主观负荷）。

| 指标 | 目标 | 结果 | 备注 |
| --- | --- | --- | --- |
| 成功率 | ≥ 90% | 92% | 11/12 完成所有任务 |
| 平均完成时间 | < 150s | 126s | 求助最快 18s |
| 错误率 | < 10% | 7% | 主要为误触求助取消 |
| SUS | ≥ 75 | 81 | 家属端 83，患者端 78 |
| NASA-TLX（低最好） | < 40 | 32 | 字体调整降低认知负荷 |

#### 6.3.1 发现的问题（节选）

| 编号 | 现象 | 影响 | 优先级 | 解决方案 |
| --- | --- | --- | --- | --- |
| U1 | 求助倒计时提示不够明显 | 可能误取消 | 高 | 增加语音提示 + 振动；倒计时数字居中放大 |
| U2 | 回忆页瀑布流加载慢 | 影响情绪体验 | 中 | 本地缓存缩略图，分段加载 |
| U3 | 家属端“临时提醒”入口在折叠区 | 降低触达 | 中 | 提升到首屏卡片区域 |
| U4 | 字体调整后部分按钮溢出 | 可访问性风险 | 高 | 使用自适配布局与 Wrap，验收 200% 放大 |
| U5 | 视频连线弱网提示不明显 | 用户误判故障 | 中 | 增加“弱网模式”文字与仅音频 fallback |

#### 6.3.2 改进迭代

- 已完成：
  - 求助倒计时 + 语音提示；
  - 字体 200% 兼容检查，按钮改为自适配；
  - 临时提醒入口上移至监控页主卡片；
  - 视频弱网提示与仅音频降级。
- 计划中：
  - 回忆页图片预加载与分块加载；
  - 社区发帖草稿自动保存；
  - 家属端“批量提醒模版”功能。

### 6.4 数据分析与结论

- **闭环改善**：第一次→第二次，SUS 提升 9 分，错误率下降 5%；绑定/提醒/求助的可发现性显著提高。
- **效率提升**：平均任务时间减少 22s；求助响应更快。
- **情感反馈**：患者端对“大按钮+语音”满意度高；回忆页带来积极情绪。
- **剩余挑战**：弱网视频、图片加载、社区活跃度仍需优化；需要更多真实患者长期试用验证。

## 7. 结论与迭代计划

- 项目实现了“机器狗 + 双端 App + 通知/监控”核心闭环，满足记忆辅助、情感陪伴、安全监护的主要需求。
- 双轮可用性评估验证了设计方向：大字号、简化流程、显性绑定、求助倒计时、通知中心等均有效降低认知负荷。
- 后续迭代重点：
  1) 性能：弱网视频与图片加载优化；
  2) 可访问性：进一步验证屏幕阅读器、色盲模式；
  3) 数据：建立长期服药/活动趋势可视化，支持医生查看；
  4) 安全：更细粒度的隐私授权与日志审计；
  5) AI 能力：本地轻量化模型，提升离线问答与物体识别。
